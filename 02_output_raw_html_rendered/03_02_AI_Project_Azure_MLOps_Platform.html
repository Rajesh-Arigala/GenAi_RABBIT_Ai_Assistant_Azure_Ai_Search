Enterprise AI Platform on Azure

End-to-End Azure MLOps Pipelines for Production ML Deployment

This project demonstrates an end-to-end MLOps workflow on Azure for automating machine learning model training, evaluation, registration, and deployment. Using taxi fare prediction as the sample business scenario, the solution shows how organizations can convert raw business data into a production-ready prediction service through Azure Machine Learning and Azure DevOps pipelines.

While the model predicts taxi fares, the real value of the project is the reusable MLOps platform behind it. The same architecture can be adapted for real-world use cases such as demand forecasting, price prediction, customer churn prediction, risk scoring, and operational decision automation.

Business Problem

Why enterprises need a repeatable ML delivery pipeline

The Problem

Many companies build machine learning models in notebooks, but struggle to move them into production. Models are often trained manually, deployed inconsistently, and difficult to monitor or update. This creates delays, quality issues, poor governance, and operational risk.

The Solution

This project solves that problem by creating a repeatable ML delivery pipeline where data preparation, model training, validation, registration, deployment, and testing are automated.

Business Architecture Story

How business data becomes a production prediction service

Architecture Overview

A business team provides historical data. The data science pipeline prepares the data, trains the model, evaluates quality, and registers the approved model. The DevOps pipeline then deploys the model as an endpoint that applications or business systems can call for predictions.

The architecture separates infrastructure, training, and deployment into independent Azure DevOps pipelines. Business data is registered in Azure ML, processed through a training pipeline, evaluated, versioned in the model registry, and then deployed through either real-time online endpoints or lower-cost batch inference depending on the business use case.

Cost-Aware Azure MLOps Architecture

FinOps thinking built into training, deployment, and infrastructure choices

Cost-Aware Design

This project was designed with cloud cost control in mind. The architecture separates training, deployment, and infrastructure pipelines so compute resources are only used when required, while production endpoints can be managed based on real business demand.

The training compute cluster uses a scale-to-zero configuration, meaning no training compute cost is incurred when the pipeline is idle. This makes the solution suitable for experimentation, scheduled retraining, and cost-controlled proof-of-concept deployments.

For production inference, the project supports both online and batch deployment patterns. Online endpoints are suitable for real-time API predictions but have a continuous running cost. Batch endpoints are more cost-effective for scheduled or offline scoring workloads where real-time response is not required.

Cost Considerations Implemented

  • Azure ML training compute configured with minimum nodes set to zero
  • Training cluster scales up only during pipeline execution
  • Dedicated online endpoint used only when real-time prediction is required
  • Batch endpoint structure included for lower-cost offline inference
  • Separate infrastructure, training, and deployment pipelines for better cost visibility
  • Workspace uses Blob Storage, Key Vault, Container Registry, and Application Insights
  • Azure DevOps pipelines can run within free-tier limits for small project usage

Estimated Cost Profile

Environment Type Approximate Monthly Cost
Demo / portfolio setup with endpoint deleted after testing $5-$40
Occasional training with workspace and storage retained $10-$50
One always-on online endpoint $100-$180
Small production-style setup $200-$500
Multi-environment setup such as dev and prod $300-$700+

For low-volume or scheduled prediction workloads, batch inference can be used to reduce cost. For real-time business applications, the online endpoint can be enabled with a clear understanding of the monthly compute cost.

Resources the Code Uses

Core services, resource settings, and main cost drivers

Resource Inventory

  • Azure Resource Group
  • Azure ML Workspace in eastus
  • Azure ML Compute Cluster: Standard_D2s_v3
  • Training cluster min nodes: 0
  • Training cluster max nodes: 4
  • Azure ML Online Endpoint
  • Online deployment instance: STANDARD_D2AS_V4
  • Instance count: 1
  • Azure ML Model Registry
  • Blob Storage, Key Vault, Container Registry, Application Insights
  • Azure DevOps pipelines

Main Cost Drivers

The biggest cost is the online endpoint because the deployment uses STANDARD_D2AS_V4 with instance_count: 1. If this runs 24/7, estimate around $70/month for VM compute alone before storage, logs, registry, and workspace support services.

Training compute is cost-controlled because the cluster is configured as training_target_min_nodes: 0, training_target_max_nodes: 4, and training_target_sku: Standard_D2s_v3. That means the training cluster can scale down to zero when idle.

Estimated Monthly Cost

Scenario Estimated Cost
Infrastructure created, no training, no endpoint running $5-$25/month
Occasional training only $10-$40/month
Training + one always-on online endpoint $85-$130/month
Training + endpoint + logs/storage growing $100-$180/month
If scaled to 2 endpoint instances $160-$260/month
Dev + prod environments $180-$350+/month

For a portfolio/demo project, keep the endpoint deleted when not testing. Then the monthly cost should stay around $5-$40. For a small production setup, a realistic range is often $200-$500/month depending on traffic, uptime, monitoring, and retraining frequency.
